Section 01

Présentation Générale du Projet

1.1 Contexte et objectifs

TimeTrack BGFIBank est une application web interne développée pour la BGFIBank afin de permettre le suivi en temps réel du temps de travail des collaborateurs. Elle répond au besoin de mesurer la contribution effective de chaque agent aux objectifs stratégiques de la banque.

L'application permet à chaque employé de saisir, démarrer et arrêter des sessions de travail associées à des tâches définies, des processus métiers et des objectifs stratégiques. Les données collectées alimentent des tableaux de bord analytiques pour l'encadrement et la direction.

Objectif principal : Mesurer et optimiser la productivité des agents en calculant le ratio entre les heures productives saisies et la capacité journalière standard de 8 heures, par agent, département et objectif stratégique.

🐬

Migration SQLite/D1 → MySQL

La base de données a été migrée de Cloudflare D1 (SQLite) vers MySQL 8.x pour répondre aux besoins de production bancaire : gestion des transactions ACID, haute disponibilité, administration centralisée et compatibilité avec les outils de reporting internes de la BGFIBank.

2.2 Architecture applicative

L'application suit un modèle monolithique backend-frontend séparé :

  • Le backend est un unique fichier src/index.tsx compilé en dist/_worker.js
  • Chaque rôle dispose de son propre fichier JavaScript frontend servi en statique
  • La communication Frontend → Backend se fait exclusivement via des appels API REST JSON
  • Le token JWT est stocké dans le localStorage du navigateur
  • La connexion MySQL utilise un pool de connexions configuré via variables d'environnement

3.3 Migration vers MySQL NOUVEAU

🐬

Pourquoi passer de SQLite/D1 à MySQL ?

SQLite (Cloudflare D1) est adapté au développement local mais présente des limites pour un usage bancaire en production : absence de gestion multi-connexions concurrentes robuste, pas d'administration distante centralisée, incompatibilité avec les outils de reporting internes. MySQL 8.x répond à toutes ces exigences.

3.4 Stratégie de Suppression Logique (Soft Delete) NOUVEAU

Principe fondamental : Aucune donnée n'est jamais physiquement supprimée de la base de données. Toute suppression est logique : le champ status passe de 'Actif' à 'Inactif'. Cela préserve l'historique complet, respecte les contraintes de clés étrangères et permet une restauration éventuelle.

Avant suppression
status = 'Actif'
→ UPDATE →
Après suppression
status = 'Inactif'
EntitéRequête SQL exécutéeEffet sur les listes
DépartementUPDATE departments SET status='Inactif', updated_at=NOW() WHERE id=?Disparaît (GET filtre WHERE status='Actif')
Objectif stratégiqueUPDATE strategic_objectives SET status='Inactif', updated_at=NOW() WHERE id=?Disparaît de la liste et des graphiques
ProcessusUPDATE processes SET status='Inactif', updated_at=NOW() WHERE id=?Disparaît (ses tâches restent en base)
TâcheUPDATE tasks SET status='Inactif', updated_at=NOW() WHERE id=?Disparaît de la liste des tâches disponibles
UtilisateurUPDATE users SET status='Inactif', updated_at=NOW() WHERE id=?Ne peut plus se connecter, disparaît de la liste
Filtre automatique sur les GET : Tous les endpoints GET /api/admin/* incluent WHERE status = 'Actif', ce qui garantit que les éléments désactivés disparaissent immédiatement de l'interface après suppression.

4.2 Jointures principales par fonctionnalité

🔐 Authentification — Récupération d'un utilisateur actif avec son département LEFT JOIN
SELECT u.*, d.name AS department_name FROM users u LEFT JOIN departments d ON u.department_id = d.id WHERE u.email = ? AND u.status = 'Actif'
📊 Rapport Admin — Sessions avec toutes les informations jointes JOIN multiple
SELECT ws.*, CONCAT(u.first_name, ' ', u.last_name) AS agent_name, d.name AS department_name, t.name AS task_name, p.name AS process_name, o.name AS objective_name FROM work_sessions ws JOIN users u ON ws.user_id = u.id JOIN departments d ON ws.department_id = d.id JOIN tasks t ON ws.task_id = t.id JOIN processes p ON t.process_id = p.id JOIN strategic_objectives o ON ws.objective_id = o.id ORDER BY ws.start_time DESC

Cette requête relie 6 tables en chaîne. Utilisée pour les rapports complets avec export CSV.

🎯 Stats Admin — Heures par objectif (3 statuts) LEFT JOIN + GROUP BY
SELECT o.name, o.color, o.target_percentage, COALESCE(SUM(ws.duration_minutes), 0) AS total_minutes FROM strategic_objectives o LEFT JOIN work_sessions ws ON ws.objective_id = o.id AND ws.status IN ('Validé', 'En attente', 'En cours') WHERE o.status = 'Actif' GROUP BY o.id, o.name, o.color ORDER BY total_minutes DESC
📈 Productivité — 3 catégories par agent (MySQL) LEFT JOIN + CASE WHEN
SELECT u.id, CONCAT(u.first_name, ' ', u.last_name) AS agent_name, COALESCE(SUM(CASE WHEN ws.status = 'Validé' THEN ws.duration_minutes ELSE 0 END), 0) AS validated_minutes, COALESCE(SUM(CASE WHEN ws.status = 'En attente' THEN ws.duration_minutes ELSE 0 END), 0) AS pending_minutes, COALESCE(SUM(CASE WHEN ws.status = 'En cours' THEN ws.duration_minutes ELSE 0 END), 0) AS inprogress_minutes FROM users u LEFT JOIN work_sessions ws ON ws.user_id = u.id AND DATE(ws.start_time) = CURDATE() WHERE u.role = 'Agent' AND u.status = 'Actif' GROUP BY u.id, u.first_name, u.last_name

Section 07 — MIS À JOUR

Calcul de Productivité (3 catégories)

7.1 Les 3 catégories de productivité NOUVEAU

Nouveau : La productivité est désormais calculée sur 3 catégories au lieu de 2, offrant une vision plus précise de l'activité journalière. La capacité standard reste 480 minutes (8h). Les weekends sont automatiquement détectés et exclus (capacité = 0).
CatégorieStatut sessionsCouleur graphiqueSignification
✅ Validéstatus = 'Validé'VertSessions approuvées par le chef de département
⏳ En attentestatus IN ('Terminé','En cours')OrangeSessions pointées mais pas encore validées
❌ Non pointé480 − (validé + en attente)RougeHeures non couvertes par une session

7.2 Formules de calcul

-- Exemple pour un agent ayant 3h validées et 2h en attente (sur 8h) validated_minutes = 180 -- 3h pending_minutes = 120 -- 2h inprogress_minutes = 0 total_pointed = MIN(180 + 120 + 0, 480) = 300 -- plafonné à 480 min non_pointed = MAX(480 - 300, 0) = 180 -- 3h non pointées productive_pct = 300 / 480 × 100 = 62.5 % -- taux global validated_pct = 180 / 480 × 100 = 37.5 % -- taux validé

7.4 Gestion des weekends NOUVEAU

📅

Exclusion automatique des jours non ouvrables

Le backend détecte si la date courante est un samedi ou dimanche via new Date().getDay(). Si weekend, la capacité est définie à 0 minute et un message spécial est affiché dans le tableau de bord. Le taux de productivité affiche automatiquement "Weekend — Pas de journée de travail". En MySQL, l'équivalent est DAYOFWEEK(NOW()) (1=dimanche, 7=samedi).

Section 09 — NOUVEAU

Code Reset — Réinitialisation Sécurisée du Mot de Passe

9.1 Pourquoi ce mécanisme ?

⚠️

Problème sans code reset

L'administrateur devait saisir un nouveau mot de passe en clair, le communiquer oralement à l'agent (risque d'écoute), et l'agent ne changeait souvent pas ce mot de passe temporaire. L'administrateur connaissait donc en permanence les mots de passe — non conforme aux bonnes pratiques bancaires.

🔐

Solution : code temporaire à usage unique

L'administrateur génère un code à 6 chiffres valable 30 minutes. L'agent utilise ce code pour définir lui-même son nouveau mot de passe. L'administrateur ne voit jamais le nouveau mot de passe. Le code est détruit après utilisation.

  2. Cliquer sur l'icône 🔑 (clé) à côté de l'agent concerné
  3. Un modal affiche le code généré, l'email cible et la durée de validité
  4. L'administrateur communique ce code à l'agent (en personne ou par téléphone)

Section 10 — NOUVEAU

Notifications Temps Réel

10.1 Principe du Polling

L'application implémente un système de notifications légères basé sur le polling HTTP (interrogation périodique). Toutes les 30 secondes, le frontend appelle la route GET /api/notifications?since=<ISO_DATE> pour récupérer les nouvelles sessions soumises par les agents depuis la dernière vérification.

🔔

Pourquoi le polling plutôt que WebSocket ?

L'environnement Cloudflare Workers/Pages ne supporte pas les connexions WebSocket persistantes. Le polling toutes les 30 secondes est le compromis optimal : faible charge serveur, latence acceptable pour un usage bancaire interne (les validations ne nécessitent pas d'alertes à la milliseconde).

Glossaire

TermeDéfinition
API RESTInterface de programmation basée sur HTTP permettant la communication entre le frontend et le backend via des ressources JSON.
AUTO_INCREMENTPropriété MySQL qui génère automatiquement un identifiant unique croissant pour chaque nouvelle ligne (équivalent AUTOINCREMENT de SQLite).
CASE WHENExpression conditionnelle SQL équivalente à un IF/ELSE, utilisée ici pour les calculs de productivité par catégorie.
Code ResetCode numérique à 6 chiffres, valable 30 minutes et à usage unique, permettant à un agent de réinitialiser son mot de passe sans passer par l'administrateur.
COALESCEFonction SQL qui retourne le premier argument non-NULL. Utilisée pour remplacer les NULL par 0 dans les sommes.
CONCATFonction MySQL pour concaténer des chaînes. Remplace l'opérateur || de SQLite. Ex : CONCAT(first_name, ' ', last_name).
CURDATE()Fonction MySQL retournant la date du jour. Remplace date('now') de SQLite.
D1Ancien service de base de données SQLite de Cloudflare utilisé en développement local. Remplacé par MySQL en production.
DATE_FORMATFonction MySQL pour formater les dates. Remplace strftime() de SQLite. Ex : DATE_FORMAT(start_time, '%Y-%m').
ENUMType de données MySQL permettant de définir une liste de valeurs acceptées. Utilisé pour status, role, session_type, task_type.
FK (Foreign Key)Clé étrangère — colonne qui référence la clé primaire d'une autre table, garantissant l'intégrité référentielle.
HonoFramework web TypeScript ultra-léger optimisé pour les environnements Edge (Cloudflare Workers, Deno, Bun).
InnoDBMoteur de stockage MySQL utilisé dans ce projet. Supporte les transactions ACID et les clés étrangères (contrairement à MyISAM).
JWT (JSON Web Token)Standard ouvert (RFC 7519) pour la transmission sécurisée d'informations entre parties sous forme de token signé.
LEFT JOINRetourne toutes les lignes de la table gauche, même si aucune correspondance n'existe dans la table droite (NULL dans ce cas).
Modal de confirmationFenêtre HTML/CSS superposée demandant une confirmation avant toute suppression. Remplace le confirm() natif bloqué dans Cloudflare Workers.
MySQLSystème de gestion de base de données relationnelle open-source. Version 8.x utilisée dans ce projet en remplacement de SQLite/D1.
PK (Primary Key)Clé primaire — identifiant unique obligatoire de chaque enregistrement dans une table.
PM2Process Manager 2 — maintient l'application active en arrière-plan avec redémarrage automatique.
Rate LimitingLimitation du nombre de tentatives de connexion (3 max) avec blocage temporaire de 2 minutes par adresse IP.
RBACRole-Based Access Control — permissions attribuées selon le rôle (Admin, Chef, Agent).
SHA-256Algorithme de hachage cryptographique produisant une empreinte de 256 bits. Irréversible.
Soft DeleteSuppression logique — au lieu de supprimer physiquement un enregistrement, on passe son champ status à 'Inactif'. Préserve l'historique et les contraintes FK.
SPASingle-Page Application — application web dont le contenu est rendu côté client sans rechargement complet de la page.
utf8mb4Jeu de caractères MySQL supportant tous les caractères Unicode y compris les émojis (4 octets max). Recommandé pour tout projet international.
WranglerCLI officiel de Cloudflare pour développer, tester et déployer des Workers et Pages.
XOROpération logique bit à bit utilisée pour l'obfuscation réversible des mots de passe (chiffrement symétrique léger pour l'affichage admin).
